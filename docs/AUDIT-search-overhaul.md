# АУДИТ: качество поиска и доставки лидов — план перестройки

> Источник: 62 адверсариально-верифицированных находки (каждая проверена против кода, severity скорректирована).
> Дата: 2026-06-11. Критерий приоритизации: «что быстрее всего заставит платящего клиента сказать, что лиды реально помогают».

---

## 1. Краткое резюме здоровья проекта

Архитектура (prompt-enhancer → warehouse-first дозирование → мультиисточниковый сбор → LLM-фильтр конкурентов → скоринг) спроектирована правильно, и каждая защита по отдельности существует. Проблема в том, что **ключевые защиты не срабатывают именно на основном пути доставки**: warehouse-first дозы (основной поток лидов после прогрева склада) полностью минуют LLM-фильтр конкурентов и теряют relevance_score из-за несовпадения имени ключа; сегменты лемматизируются в неграмотный русский ещё при сохранении и в таком виде уходят в поисковые запросы; echo-stripper удаляет именно те типы покупателей, которые клиент явно назвал в промпте. Итог: точность и ранжирование, ради которых всё построено, на боевом пути работают в лучшем случае частично.

Вторая системная тема — **«самоокирпичивание» сбора**: несохраняемые rusprofile-строки (имя без контактов) навсегда забивают дозу, ложный штамп «всё собрано» ставится при любом сбое источников, веб-находки с city="" невидимы для гео-фильтра пере-выборки. Клиент видит «источники исчерпаны» там, где система просто перестала пытаться. Третья тема — **контакты**: телефоны в локальном формате и форматированные tel: не извлекаются, скрейп 2GIS может приклеить чужой телефон, обогащение вечно пере-скрейпит мёртвые лиды вместо свежих. При этом 2GIS API не отдаёт контакты вовсе, так что каждый из этих багов бьёт по единственному пути к email/телефону.

Хорошая новость: большинство P0-фиксов — точечные (одна строка до пары десятков строк), без миграций и переписываний. Полный список из ~25 первых шагов — в разделе 5. Отдельно: **до активации триального ключа Яндекса обязателен фикс дублирования запросов (24x burn), иначе триал сгорит за несколько поисков.**

## 2. ПОИСК — ЯДРО

Формат: **Проблема** (file:line) → **Влияние на клиента** → **Фикс** → Усилия (S/M/L) / Приоритет (P0/P1/P2).

### 2.1 Prompt-enhancer (вход воронки)

#### 2.1.1 Echo-stripper удаляет типы покупателей, явно названные пользователем — CRITICAL
- **Проблема**: `_extract_prompt_product_words` считает «словами продукта» ВСЕ не-стоп-слова промпта, включая покупателей после «для». `prompt_enhancer.py:690-737, 1039-1047`. Эмпирически: «Продаю корм для птицефабрик» → сегмент «птицефабрика» удаляется; «клининговые услуги для ресторанов и отелей» → теряются «ресторан», «отель»; префикс «клини» убивает 2GIS-категорию «клиника» (substring на :628); города в косвенных падежах («в Казани») просачиваются в product_words.
- **Влияние**: самый точный таргетинг-сигнал — то, что клиент сам напечатал — систематически исключается из всех запросов 2GIS/Yandex/SearXNG и из скоринга по сегментам. Клиент с «корм для птицефабрик» получает любые фермы, кроме птицефабрик.
- **Фикс**: разделить промпт на SELL/BUY-клаузы: слова после «для», «клиенты», «поставляем в» — слова ПОКУПАТЕЛЯ, исключить из product_words и занести в whitelist для `_strip_product_echoes`. Substring-фильтр на :628 → сравнение лемм по границе слова, мин. длина стема 6+. Города стрипать по pymorphy-нормальформам. Ещё лучше: пусть LLM возвращает product_words/customer_words отдельными полями, лексика — только fallback.
- **M / P0**

#### 2.1.2 Лемматизация сегментов при сохранении ломает запросы и фильтры — HIGH (объединяет 2 находки)
- **Проблема**: `_normalize_segments` лемматизирует каждое слово (`prompt_enhancer.py:43-54, 1037-1038`): «строительные компании»→«строительный компания», «управляющие компании ЖКХ»→«управлять компания жкх». Эти строки сохраняются в project.segments и используются ДОСЛОВНО: quoted-запросы SearXNG `f'"{seg}" "{geo}" ООО'` (lead_collection.py:687), термы 2GIS/Yandex (:2093, 804-809), rusprofile (:1769). Они же ломают whole-phrase keep-сигнал rule-фильтра (llm_filter.py:503 — substring по сырому тексту) и ключи `_SEGMENT_OKVED_MAP`. Лемматизация при хранении избыточна: матчеры (`_build_match_terms`, lead_collection.py:241-275) лемматизируют сами.
- **Влияние**: каждый LLM-проект ищет грамматически сломанные типы покупателей: quoted-запросы дают ~0 результатов, фразовый keep-сигнал мёртв, в UI клиент видит ломаный русский.
- **Фикс**: в `_try_llm_enhance` хранить сегменты в натуральной форме, дедуп — по лемма-ключу (`seen.add(_lemmatize_phrase(seg))`, append оригинал). Бэкфилл сегментов существующих проектов повторным enhance. Обновить тест `test_normalize_segments_dedupes_and_lemmatizes`.
- **S (+M бэкфилл) / P0**

#### 2.1.3 Rule-fallback маппит промпт в чужую отрасль по голому substring — HIGH
- **Проблема**: `_smart_fallback` матчит `if keyword in prompt_lower` без границ слов (`prompt_enhancer.py:1081`): «колеса и шины» → лесо-/дачное строительство (через «леса»), «брусчатка» → «дома из бруса». Заявленный в комменте 2-keyword-минимум для HoReCa (:77) не реализован.
- **Влияние**: при любом сбое LLM проект шиномонтажника навсегда (кэш в jobs.py:178-186) наполняется сегментами дачных строителей.
- **Фикс**: word-start lookbehind `(?<![а-яёa-z])` + для коротких стемов («брус», «сто») — точное слово/лемма или longest-match конкуренция (lookbehind НЕ закрывает «брусчатку» — слово начинается с «брус»). Реализовать 2-keyword-минимум HoReCa (оборудование И заведение).
- **S / P1**

#### 2.1.4 OKVED-fallback выдаёт абсурдные коды — MEDIUM
- **Проблема**: `_okved_from_segments` — substring без границ (`prompt_enhancer.py:886-901`): «столовая»/«стоматология»/«ресторан» → 45.20 «авторемонт» (ключ «сто», :831); «банкетный зал» → 64.19 «банки» (ключ «банк», :845). Conf 0.9, первым в списке.
- **Влияние**: на карточке проекта клиент видит «ОКВЭД 45.20 авторемонт» при поиске столовых — выглядит сломанным, подрывает доверие ко всему остальному.
- **Фикс**: проверено: `r'\bсто\b'`, `r'\bбанк(?!етн)'`, общий lookbehind для остальных ключей; то же для «тц»/«бц» (:796).
- **S / P1**

#### 2.1.5 Fallback неотличим от LLM-результата и кэшируется навсегда; generic-fallback эхоит продукт как нишу — MEDIUM
- **Проблема**: `enhance_prompt` возвращает одинаковый dict с обоих путей (`prompt_enhancer.py:494-528`); jobs.py:178-191 кэширует search_query на первом же вызове и никогда не пересчитывает. Generic-fallback ставит `search_queries_niche = raw_prompt[:120]` (:1122) — текст ПРОДУКТА клиента становится нишей для warehouse-матчинга и скоринга (буст конкурентам).
- **Влияние**: один сбой LLM при создании/первом сборе навсегда запирает проект на rule-based термах даже после восстановления LLM.
- **Фикс**: `result["source"] = "llm"|"fallback"|"fallback_generic"`; в jobs.py кэшировать search_query только при source=="llm" (fallback ретраится на следующей дозе); в generic-fallback не эмитить raw_prompt как нишу (учесть, что поле `niche` там тоже = raw_prompt — обработать оба).
- **S / P1**

#### 2.1.6 Гео-экстракция не знает целые крупные города → вся Россия — MEDIUM
- **Проблема**: `_extract_geography` знает ~60 городов (`prompt_enhancer.py:1128-1142`): «в Ставрополе», «в Севастополе», «в Грозном», «в Наб. Челнах» → «Россия» → nationwide fan-out на 16 чужих городов. Несогласованность: «Ставропольский край» резолвится, сам город — нет.
- **Влияние**: fallback-клиент локальных услуг получает лиды из Москвы/СПб вместо своего города — вся пачка бесполезна.
- **Фикс**: смержить values region_to_city в cities, расширить до городов >100k (или взять ключи `_CITY_SLUG_MAP`), и **нормализовать pymorphy обе стороны** (вход И записи списка) — иначе «Набережных Челнах» так и не матчится (проверено).
- **S / P1**

### 2.2 Генерация запросов и оркестрация источников

#### 2.2.1 Yandex Maps API: идентичный список запросов гоняется до 24 раз за город — HIGH (объединяет 2 находки; КРИТИЧНО до активации ключа)
- **Проблема**: `_search_yandex_maps` вызывается внутри цикла по термам (`lead_collection.py:2137, 2190-2193`), но в buyer-hunt режиме `_build_yandex_map_queries` (:802-809) игнорирует term и строит запросы ТОЛЬКО из segments[:24] — все 24 итерации перезапускают идентичный список, ранний break по per_term_limit гарантирует, что реально опрашивается только первый сегмент. Bbox-геокодинг повторяется на каждый вызов (:959): до 384 платных вызовов на nationwide-проект.
- **Влияние**: триальный 14-дневный ключ сгорит ~в 24 раза быстрее на дублях (риск перманентного `_YANDEX_DEAD_KEY`); сегменты после первого получают НОЛЬ покрытия от источника с максимальным весом (64); дубли забивают oversample-кап.
- **Фикс**: вынести вызов из цикла — один вызов на map_geo с полным списком сегментов и бюджетом = остаток oversample (равномерно по сегментам), либо `segments=[term]` на итерацию. Кэшировать bbox по geo (dict/Redis). Сделать ДО включения ключа.
- **S / P0**

#### 2.2.2 Oversample-кап считает дубли между источниками — веб-проход (единственный источник сайтов) голодает — HIGH
- **Проблема**: `collect_candidates` аппендит каждую строку без кросс-источникового дедупа (`lead_collection.py:2108-2117`); карты+rusprofile идут строго ДО SearXNG, и :2227 брейкает веб-цикл при заполненном капе. В плотных нишах maps-фан-аут заполняет кап в одиночку — ноль веб-запросов. А сайты/email входят в пайплайн только через веб-строки.
- **Влияние**: дозы по популярным нишам в больших городах (ядро спроса) — name+address-строки без сайтов → без email, скошенные к первым сегментам. «Экспорт 2GIS без телефонов».
- **Фикс**: считать к капу уникальные dedup-ключи (get_base_domain | name|city), дубли по-прежнему аппендить (нужны для merge полей в `_finalize_candidates`). Зарезервировать явный бюджет веб-прохода: гонять SearXNG до min(limit, N) веб-кандидатов независимо от объёма карт; тот же резерв для Bing-гейта (:2255).
- **M / P0**

#### 2.2.3 Rusprofile получает фейковый город (= запрошенное гео) — гео-гард и гео-скоринг обмануты — HIGH
- **Проблема**: `_search_rusprofile` парсит только id+имя, но эмитит `"city": geo.strip()` (`lead_collection.py:1823`). Hard geo guard (:502-506) и geo_hits (:508-517) всегда проходят — внерегиональные юрлица не дисквалифицируемы и получают гео-бонус вместо штрафа -30.
- **Влияние**: клиент со скоупом «Томск» обзванивает московское ООО с пометкой «Томск»; фейковый город пишется в кросс-орговый склад и отравляет все будущие дозы всех организаций по этой нише+гео.
- **Фикс**: парсить реальный регион из сниппета выдачи rusprofile рядом с анкором (проверить вёрстку с прод-сервера — с этой машины rusprofile отдаёт 404-шелл). Просто `city=""` уронит каждую строку ниже порога 26 и убьёт источник — тогда придётся смягчать не-map гео-штраф для registry-строк при отсутствии контр-улик.
- **M / P0**

#### 2.2.4 Кандидаты обрезаются до `limit` ДО LLM-фильтра, без добора — MEDIUM
- **Проблема**: `ranked = _finalize_candidates(collected, limit)` (lead_collection.py:2281) до `filter_candidates_llm` (:2284); 5x-оверсэмпл выбрасывается, отфильтрованные конкуренты не замещаются.
- **Влияние**: систематический недобор лидов на поиск + меньше посева склада за те же деньги.
- **Фикс**: финализировать с буфером `limit*2` (или чанками по `limit` до набора выживших), срез `[:limit]` ПОСЛЕ фильтра (оба фильтра сохраняют порядок). Применить во всех гео-тирах.
- **S / P1**

#### 2.2.5 Гео-каскад: tier-2 просит только `remaining`, дедуп — после — MEDIUM
- **Проблема**: tier-2 вызывается с `remaining` (lead_collection.py:2042-2048), топ-слоты съедают уже известные компании; `_city_to_slug` (:1428-1430) маппит «Томская область»→'tomsk' — те же страницы 2GIS; кросс-тир дедуп по сырому domain (а не base_domain); лог «added %d new» врёт (pre-dedup).
- **Влияние**: спасательный проход для тонких ниш платит второй мультиисточниковый поиск + второй LLM-pass и часто добавляет ~0.
- **Фикс**: передавать в tier-2 exclusion-set уже виденных base-domain/name|city ключей (применять при сборе, до трюнка); дедуп через get_base_domain; логировать post-dedup дельту.
- **M / P1**

#### 2.2.6 SearXNG-запросы 45-65 слов — селлер-негативы за капом движков — MEDIUM
- **Проблема**: 40 CORE + 20 SELLER негативов, SELLER в конце (`lead_collection.py:183-218, 684-687`); Google режет после 32 слов — все 20 анти-конкурентных негативов инертны.
- **Фикс**: сократить до <=10 токенов, селлер-негативы ПЕРВЫМИ; доменные негативы убрать (уже фильтруются post-fetch через is_aggregator_domain); сохранить условный сплит `_pick_negatives` (проекты, ищущие магазины); то же для Bing (:2263).
- **S / P1**

#### 2.2.7 `_pick_negatives`: один сегмент с «интернет» отключает селлер-негативы всему проекту — MEDIUM
- **Проблема**: решение по блобу всех сегментов один раз (`lead_collection.py:204-218`, вызов :672); «интернет-провайдер» (легитимный покупатель) гасит исключение конкурентов для всех 24 сегментов.
- **Фикс**: решать пер-сегмент (`_pick_negatives([seg])` в цикле :681 и в Bing :2259); сузить «интернет» → «интернет-магазин» (голый «магазин» уже покрывает).
- **S / P1**

#### 2.2.8 segments[:24] молча отбрасывает до 16 из 25-40 сегментов — MEDIUM
- **Проблема**: энхансер требует 25-40 сегментов, все query-билдеры режут [:24] (`lead_collection.py:681, 804, 818, 2093`; склад — company_warehouse.py:380), Bing — [:3]. Сегменты 24+ никогда не ищутся, окно детерминировано.

#### 2.2.9 ОКВЭД-коды извлекаются, но не используются ни одним query-путём — MEDIUM
- **Проблема**: project.okved_codes — только UI; rusprofile ищется free-text'ом `f"{niche} {geo}"` (lead_collection.py:1769) — находит только компании, НАЗВАННЫЕ как сегмент, и пропускает тысячи «ООО Заря» с ОКВЭД 01.41.

#### 2.2.10 Мелочь (P2, S каждая)

### 2.3 Релевантность и конкурент-фильтр (llm_filter)

#### 2.3.1 Парсер ответа LLM выгребает ВСЕ цифры — отказы превращаются в keep — HIGH
- **Проблема**: `re.findall(r"\d+", answer)` (`llm_filter.py:295-311`): «Подходят 1, 3. Кандидаты 2 и 4 — конкуренты» сохраняет и 2, и 4; «Подходящих: 0 из 30» ломает all-rejected-эвристику и оставляет №30; «1-5» теряет 2-4. Неверные вердикты кэшируются на 7 дней.
- **Влияние**: явно отклонённые LLM конкуренты доставляются клиенту и замораживаются в Redis на неделю — «почему я вижу своего конкурента в лидах».
- **Фикс**: структурный вывод `{"keep":[...]}` + json.loads; fallback — цифры только после последнего «ПОДХОДЯЩИЕ»/из последней строки; ответ с цифрами И токенами отказа («конкурент», «не подход», «0 из») → parse failure → rule-fallback; диапазоны разворачивать или фейлить. `{"keep":[]}` = all-rejected. **Бампнуть префикс кэша llmf:→llmf2:** (config-hash не включает шаблон — отравленные вердикты иначе доживут неделю).
- **M / P0**

#### 2.3.2 Кандидаты, выпавшие из обрезанного ответа, кэшируются как drop на 7 дней — HIGH
- **Проблема**: BATCH_SIZE=30 при max_tokens=200 (`llm_filter.py:180, 281`); усечение неопределимо (llm_client отбрасывает stop_reason/status), парс «успешен», и :200-207 кэшируют всех неназванных как "0" на 7 дней.
- **Влияние**: один обрезанный ответ молча выпиливает хвост батча из выдачи И из всех будущих прогонов недели.
- **Фикс**: max_tokens ~500; кэшировать "0" только для ЯВНЫХ drop (или вовсе не кэшировать omissions); детерминированно — пробрасывать сигнал усечения из llm_client (Yandex ALTERNATIVE_STATUS_TRUNCATED_FINAL, Anthropic stop_reason, GigaChat finish_reason) и фейлить батч.
- **S-M / P0**

#### 2.3.3 Rule-fallback извлекает слова ПОКУПАТЕЛЕЙ как «термы продукта» и режет целевых клиентов как конкурентов — HIGH
- **Проблема**: «Продаю овощи для ресторанов» → core terms ['свежие','рестор'] (`llm_filter.py:391-426`); Step 1 (:486-493) отклоняет «Ресторан Пушкин» (имя+«заказать»/«доставка по» — почти универсальные маркеры). Синтез-промпт `{niche} для {segments}` (:94-107) кормит экстрактор чистой аудиторией — kept=[] (воспроизведено).
- **Влияние**: при недоступном/закэпленном LLM (штатное состояние) дозы выпотрошены именно по запрошенным сегментам.
- **Фикс**: вычитать словарь segments+niche из core_terms перед Step 1 (проверено: чинит инверсию, конкуренты по-прежнему режутся); резать промпт на первом « для »; при синтезированном промпте Step 1 пропускать вовсе; ту же вычистку — в `_extract_product_keywords` (Step 2).
- **S / P0**

#### 2.3.4 Warehouse-доза полностью минует filter_candidates_llm — HIGH
- **Проблема**: единственный вызов фильтра — внутри search_leads (live). Дозы из склада (`jobs.py:230-242, 282-292`) идут в проект без buyer-vs-competitor проверки, хотя склад кросс-орговый и docstring search_warehouse прямо обещает «recall over precision, since the live pipeline re-scores everything afterwards» — что на этом пути ложь.
- **Влияние**: главный рычаг точности (BUYER-HUNT) выключен на пути, который обслуживает большинство лидов после прогрева склада: клиент Б получает конкурентов, прошедших только фильтр клиента А.

- **S-M / P0**

#### 2.3.5 «для» как продукт-keyword схлопывает порог Step 2 — MEDIUM

### 2.4 Скоринг

#### 2.4.1 Лиды со склада всегда скорятся с relevance_score=0 — несовпадение имени ключа — HIGH (был critical; фикс = 1 строка)
- **Проблема**: `_company_to_candidate` возвращает релевантность как `"score"` (`company_warehouse.py:453`), а save-loop читает `c.get("relevance_score", 0)` (`jobs.py:386, 402`). Весь основной путь скорится с нулевой релевантностью; префикс `relevance=NN;` в notes не пишется → enrich-рескор (:737-751) фиксирует 0 навсегда.
- **Влияние**: единственная компонента скора, несущая поисковую релевантность (вес источника, niche/geo-матч, штраф конкурента), выбрасывается почти для всех видимых клиенту лидов.
- **Фикс**: добавить `"relevance_score": int(row.best_score or 0)` в `_company_to_candidate` (шкала совпадает — best_score хранится из relevance_score при upsert).
- **S / P0**

#### 2.4.2 Полные контакты без какого-либо таргет-сигнала = 83 = «hot» — HIGH
- **Проблема**: base(35)+domain(10)+email(20)+phone(10)+address(8)=83 ≥ 80 («hot» в UI) при нуле keyword/segment/relevance (`scoring.py:233-242`); воспроизведено.
- **Влияние**: сортировка по скору показывает «контактных, но мимо цели» выше настоящих покупателей — первое впечатление о качестве убито.
- **Фикс**: `if match_pts == 0: score = min(score, 70)` перед _clamp; матчить keyword/segment ещё и по snippet/notes (склад-строки несут релевантность в description). Долгосрочно: base→~20, relevance-полоса→~30.
- **S / P0**

#### 2.4.3 keyword_bonus награждает сигнатуру конкурента: «ТД Кормовые Добавки Оптом»=95 vs покупатель=43 — HIGH
- **Проблема**: +12 за слова ниши (= продукта клиента) в имени/домене (`scoring.py:256-261`); склад вдобавок ILIKE-матчит normalized_name по нише — активно предпочитает селлеров.
- **Фикс**: селлер-маркеры в score_lead (копия `_COMPETITOR_SIGNALS`, НЕ импорт тяжёлого lead_collection): при segments и маркере в имени — скип бонуса и -15. Гард как в `_pick_negatives`: если сами сегменты — селлер-категории («магазины»...), штраф не применять.
- **S / P0**

#### 2.4.4 Адрес считается «контактом» — недосягаемые 2GIS-лиды без штрафа — MEDIUM
- **Проблема**: `has_any_contact = email or phone or address` (`scoring.py:244-246`); address-only лиды (норма для 2GIS) доходят до amber/mint.
- **Фикс** (проверен, 59 тестов проходят): `has_any_contact = has_email or has_phone or bool(domain)`; address оставить как +8; опционально cap min(score, 65) без email/phone/domain. Обновить докстринг и коммент в config.py:56.

### 2.5 Складское дозирование (warehouse-first)

#### 2.5.1 Несохраняемые rusprofile-строки навсегда забивают дозу и «окирпичивают» сбор с ложным «всё собрано» — CRITICAL
- **Проблема**: `_take()` (`jobs.py:217-228`) проверяет только новизну ключа; rusprofile-строки (только имя, без domain/phone/address — lead_collection.py:1821-1834) никогда не проходят save-loop (:353-361), но пишутся в склад (:277) и пере-сервятся каждый прогон в порядке best_score DESC. Когда таких строк ≥ batch_size: доза «полна», deficit-гейт (:249) ложно False, live-поиск не запускается, added==0, клиенту — «всё доступное уже собрано» (:463-467). Стационарное состояние навсегда.
- **Влияние**: проект застревает на 0 новых лидов за прогон, продукт врёт об исчерпании. Ударит большинство проектов со 2-го прогона.
- **Фикс**: (1) в `_take()` зеркалировать правила сохраняемости (имя; домен через is_real_domain; домен-less без phone И address — скип); (2) сделать rusprofile сохраняемым: добавить 'rusprofile' в is_maps (:353) и снять контакт-требование (:360) при наличии rusprofile_id — enrich имеет 2GIS-by-name fallback (:646-657). **Важно: (1) в одиночку чинит брик только если НЕ исключать rusprofile_id-строки из скипа; правильный комплект — (1)+(2).**
- **M / P0**

#### 2.5.2 Свежие веб-строки (city="") невидимы для гео-фильтрованной пере-выборки; stored>0 глушит raw-live fallback — HIGH
- **Проблема**: searxng/bing-кандидаты без города (lead_collection.py:652, 756) → склад с пустыми city/region/address → гео-предикат пере-выборки (company_warehouse.py:389-402) их не матчит никогда; fallback `_take(live)` гейтится `stored == 0` (`jobs.py:297-298`) — а веб-строки апсертнулись. Платный поиск нашёл → клиент получил ноль; + ложный exhausted-штамп.
- **Влияние**: подавляется доставка ИМЕННО источника с сайтами/email; для prompt-проектов без сегментов — перманентный ноль (смежный баг: при segments=[] SearXNG-запросов вообще нет — закрыть отдельно).
- **Фикс**: jobs.py:297 → `if len(candidates) < batch_size: _take(live)` (строго безопасно, дедуп через chosen); при write-through бэкфиллить city=effective_geo на пустых city (кроме _NATIONWIDE_GEOS).
- **S / P0**

#### 2.5.3 Штамп исчерпания не отличает аутедж от исчерпания — 12h ложный кулдаун — HIGH
- **Проблема**: `did_live and added==0` (`jobs.py:456-457`); search_leads глотает все ошибки и отдаёт [] при полном аутедже — неотличимо от «всё собрано». live_count посчитан (:273), но не используется. Смена ниши/гео кулдаун НЕ сбрасывает.

Уточнение: live_count пост-фильтровый — лучше пробросить из search_leads сырой pre-filter счётчик и штамповать по нему.
- **S / P0**

#### 2.5.4 Двойная идентичность (domain vs name|city) — одна компания продаётся дважды — MEDIUM
- **Проблема**: `_dedup_key` (company_warehouse.py:46-65): 2GIS-строка = name|city, веб-строка той же фирмы = base_domain; ни склад, ни `_take`, ни save-loop (jobs.py:364-374), ни already_keys их не связывают.
- **Влияние**: «я плачу за лида, а вы продали одну компанию дважды» + двойная квота.
- **Фикс**: `_project_dedup_keys` эмитит ОБА ключа на лид; `_take` проверяет оба; в save-loop для domain-кандидатов — доп. сверка по нормализованным name+city (Python-side: C-locale не кейс-фолдит кириллицу); долгосрочно — merge alias-строки при upsert domain-keyed компании.
